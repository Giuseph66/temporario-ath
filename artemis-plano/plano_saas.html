⬡ Minha opinião — arquitetura geral

O projeto atual tem uma base sólida e pensada. A grande mudança para SaaS não é técnica no sentido difícil — é de escopo de dados. Cada tabela que existe hoje (User, ChatHistory) precisa pertencer a um Tenant. As configs que estão em JSON fixos precisam virar linhas de banco de dados, uma por cliente.

O maior risco aqui é fazer a migração pela metade: chegar num estado onde o código tem suporte a multi-tenant mas o banco ainda é pensado para um só cliente. A ordem correta é: banco primeiro, depois API, depois frontend.

Sobre canais WhatsApp: use a Evolution API para o produto SaaS (cada cliente conecta seu próprio número via QR Code) e mantenha a Meta API como opção premium para quem tem WhatsApp Business verificado. Os dois não se excluem.

Evolution API — Integração
Você já tem uma VPS com Evolution rodando. O plano abaixo descreve como integrar ao Artemis mantendo a arquitetura atual e adicionando suporte multi-instância por tenant.

Conexão de instância por tenant

Cada cliente cria uma instância na Evolution com um nome único (artemis-{tenantId}). O backend faz chamadas REST para a API da Evolution passando a instância correta por request.

Webhook unificado no backend

A Evolution envia todos os eventos para POST /webhook/evolution com o campo instance no payload. O controller resolve o tenant a partir desse campo e processa normalmente.

QR Code de conexão via painel

O frontend exibe um QR Code gerado pela Evolution API em tempo real. Quando o cliente escaneia, a instância fica conectada e o bot começa a funcionar. Zero burocracia.

Envio de mensagens

Substituir WhatsAppService por um EvolutionService que chama POST /message/sendText/{instance}. A interface de chamada no BotService não muda.

Risco calculado: A Evolution API usa o protocolo WhatsApp Web (não oficial). Contas podem ser banidas em caso de comportamento suspeito (volume muito alto, spam, scraping). Para mitigar: limite de mensagens por hora por instância, warm-up de números novos, e nunca enviar para contatos que nunca iniciaram conversa com o número. Para clientes que precisam de garantia corporativa, ofereça a Meta API como plano premium.
⬡ Minha recomendação — estratégia de canal

Curto prazo (MVP SaaS): use apenas a Evolution API. O onboarding por QR Code é o que vai fazer o produto funcionar de verdade — nenhum cliente vai esperar semanas para aprovar uma conta Meta só para testar seu produto.

Médio prazo: crie uma abstração MessagingProvider no backend que pode usar Evolution ou Meta API. Cada tenant escolhe qual quer. Assim você não está preso a nenhum dos dois.

Sobre a sua VPS com Evolution: perfeito para começar. Quando o produto crescer, cada cliente poderá ter a Evolution rodando no próprio servidor deles — você só precisa do endpoint e da API Key deles para conectar.

sobre o projeto

⬡ O que está muito certo

O funil de vendas no bot é o ativo mais valioso. A lógica de 6 estados FSM, os prompts por estado em XML, o tratamento de objeções em 3 camadas — isso é produto, não é código. É o que diferencia o Artemis de um chatbot genérico. Não mexa nisso sem razão.

A persona no config/persona.json é extremamente bem elaborada. As restrições absolutas, o protocolo teen, a transferência para humano com 4 tipos distintos — isso tudo precisa virar configurável por tenant no painel. É o coração do produto SaaS.

⬡ O que vai ser o diferencial competitivo

A maioria dos "bots de WhatsApp com IA" no mercado brasileiro são fluxos lineares com respostas fixas. O Artemis tem memória persistente, FSM inteligente, tool calling real (Asaas + Calendar) e persona configurável. Isso é raro. Quando empacotar isso com onboarding por QR Code, a proposta de valor é clara: seu bot de vendas com IA, configurado em 10 minutos, conectado ao seu WhatsApp.

O design das telas de exemplo também é um diferencial de posicionamento — é sofisticado, editorial, não parece um painel de bot barato. Isso importa para o cliente final perceber valor premium.

⬡ O que eu priorizaria

Primeiro: feche os 23 passos do plano original (estabilização, segurança, testes). Sistema instável não escala — você vai debugar em produção com clientes reais olhando.

Segundo: Sprint A (banco) → Sprint B (auth + UI shell). Aqui você já tem painel com login. Terceiro: Sprint C (Evolution + QR Code na tela). Aqui você conecta WhatsApp pelo próprio painel. A partir desse ponto cada sprint entrega uma tela nova com dados reais — sem esperar o frontend inteiro pronto para ver alguma coisa.

Não faça: monetizar/billing antes de ter 3 clientes pagantes. Valide primeiro que o produto funciona para mais de uma empresa.

⬡ Sobre a Evolution API

É a escolha certa para o MVP SaaS. O fato de você já ter uma VPS rodando é uma vantagem enorme — você não precisa convencer clientes a pagar pela Meta API para começar a usar. O risco de ban existe mas é gerenciável com volume razoável e bom comportamento. Quando tiver clientes corporativos que precisam de garantia, você já terá a abstração MessagingProvider pronta para plugar a Meta API sem reescrever nada.
